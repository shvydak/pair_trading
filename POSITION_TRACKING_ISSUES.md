# Position Tracking: архитектурные проблемы и план решения

## Корневая проблема

Биржа ведёт учёт **по символам**, стратегия ведёт учёт **по парам**.
Это два разных пространства — текущий код смешивает их в нескольких местах.

---

## Bug 1 — Неверное направление при закрытии (простой фикс, сделать в первую очередь)

**Файлы:** `backend/main.py:372-373` и `main.py:1966-1967`

**Проблема:** при закрытии позиции код смотрит на биржу чтобы определить направление ордера:
```python
side1 = "sell" if (p1 and p1["side"] == "long") else "buy"
```
Если `p1 is None` (нога отсутствует на бирже — например, закрыта вручную) — код делает `buy` вместо `sell`, то есть **открывает новую позицию** вместо закрытия.

**Фикс:** всегда читать направление из DB `pos["side"]`:
- `long_spread` → закрытие: sell leg1, buy leg2
- `short_spread` → закрытие: buy leg1, sell leg2

Это работает независимо от состояния биржи.

## Issue — Пересекающиеся символы (сложнее, требует обдумывания)

**Пример:** ETH/BTC и LTC/BTC открыты одновременно → на бирже BTC = нетто от обоих пар.

### Все сценарии где ломается

**1. Неверное qty при закрытии одной пары**
При закрытии ETH/BTC код видит биржевой BTC qty = 0.07 (сумма обоих пар), использует его, и случайно закрывает часть LTC/BTC позиции.
- Файл: `main.py:372` (`_do_market_close`), `main.py:1699` (smart close)

**2. Ручное вмешательство → Bug 1**
Закрыл одну ногу вручную → DB думает позиция открыта → TP/SL срабатывает → код не находит ногу → неверное направление (Bug 1).

**3. Enriched positions — двойной PnL**
`/api/db/positions/enriched` матчит DB-записи к биржевым позициям **по символу**.
Если BTC в двух парах — обе DB-записи получают PnL от одного и того же биржевого BTC. PnL показан дважды, причём неверно.

**4. TP/SL smart close — та же qty проблема**
`_do_smart_close_trigger` тоже берёт qty с биржи (`abs(p1["size"])`). Та же проблема что в пункте 1.
- Файл: `main.py:413-436`

**5. Одновременное срабатывание TP на обеих парах**
`closing_pairs` защищает от двойного закрытия одной пары, но не от одновременного закрытия двух разных пар с общим символом → over-close общего символа.

**6. Усреднение (pyramiding) — нет поддержки**
Если добавить к позиции вручную — DB qty устарел, биржевой qty больше. Система не знает что позиция была увеличена.

---

## Архитектурное решение

> **DB — единственный источник правды для qty и direction. Биржа — только для цен и PnL.**

| Что | Сейчас | Правильно |
|---|---|---|
| Направление закрытия | с биржи | из DB `side` |
| Количество для закрытия | с биржи (если есть) | из DB `qty1/qty2` |
| Цена для PnL | с биржи (mark price) | с биржи (правильно) |
| Проверка расхождения | нет | предупреждение если биржа ≠ DB |

### Enriched positions
Матчинг по символу надо убрать. Биржевые позиции показывать отдельно, без соединения с конкретной парой.

---

## Предлагаемый порядок фиксов

1. **Bug 1 — направление из DB** — просто, безопасно, независимо от остального
2. **Qty из DB для всех close операций** — убирает проблему пересечений и ручного вмешательства
3. **Enriched positions** — переработать матчинг, убрать double-PnL
4. **Защита от over-close** — проверять пересечение символов при одновременном срабатывании TP/SL
5. **Предупреждение при открытии** — если символ уже есть в другой открытой паре
6. **Усреднение** — отдельная большая задача, требует изменений в DB схеме
